# 纯前端项目展示增强设计

## 背景

当前站点已经具备首页氛围营造、主题切换、PWA、赛季回顾等基础展示能力，但“产品核心概念”和“交互逻辑”仍主要依赖静态文案与卡片式内容。对于首版展示，最缺的是一个无需后端也能直观说明“这支车队如何完成无人驾驶赛车闭环”的交互式前端模块。

## 目标

1. 在首页新增一个高辨识度、强交互的展示模块，直接呈现无人驾驶赛车的核心产品概念。
2. 完全基于本地静态数据和浏览器能力实现，不依赖任何后端接口或第三方实时服务。
3. 提供可操作的交互逻辑，包括场景切换、系统状态变化、数据可视化与本地持久化。
4. 保持与现有 Astro/Starlight 首页视觉语言一致，兼顾移动端与离线可用性。

## 非目标

1. 不实现真实遥测接入、账号系统、多人协作或云端同步。
2. 不引入新的前端框架或复杂状态管理库。
3. 不重做整站首页结构，只在现有首页中补充一块高价值展示区。

## 现状判断

1. 首页已有 Hero、数据卡、倒计时、赛季回顾、主题色和粒子背景，视觉基础充足。
2. 项目已具备 `safeGetItem` / `safeSetJSON` 等本地存储能力，可直接复用。
3. 项目已有 PWA 与离线页面，因此新增模块只需坚持本地数据驱动即可自然支持离线浏览。
4. 现有展示更偏“信息陈列”，缺少“交互式系统演示”这一层。

## 方案对比

### 方案 A：只增强首页现有卡片与动画

- 优点：修改小，风险低。
- 缺点：信息密度提升有限，无法真正展示无人驾驶系统闭环。

### 方案 B：新增首页“智能驾驶交互实验室”模块

- 优点：直接放在首页黄金位置，既能保持流量入口，又能展示核心概念与交互逻辑。
- 优点：纯前端可落地，数据、状态和可视化都能本地完成。
- 缺点：需要增加一套新数据模型、交互脚本与测试。

### 方案 C：独立做一个完整 mini app 子页面

- 优点：沉浸感最强，扩展空间大。
- 缺点：首版成本偏高，首页导流成本也更高，超出“第一版展示增强”必要范围。

## 推荐方案

选择方案 B：在首页新增“智能驾驶交互实验室”模块，并通过本地数据驱动实现系统链路演示、场景切换、遥测可视化和状态持久化。

## 功能规格

### 1. 智能驾驶交互实验室

模块名称暂定为“Autonomous Showcase Lab / 智能驾驶交互实验室”。

用户可在模块中切换预设场景，例如：

1. 发车校准
2. 高速循迹
3. 八字绕环
4. 紧急制动

每个场景包含以下纯前端表现：

1. 赛道缩略图与当前车辆位置/路径高亮。
2. 遥测指标卡片，例如速度、锥桶识别率、定位置信度、控制延迟。
3. 简易 SVG 趋势图，展示一段“圈速/稳定性”变化。
4. 系统链路状态区，展示感知、定位、规划、控制、执行器五个阶段的状态说明。
5. 场景摘要与当前策略说明，帮助非技术用户理解交互结果。

### 2. 本地持久化

使用 `localStorage` 保存以下状态：

1. 用户上次查看的场景
2. 用户标记的关注子系统

用户刷新后仍能保留上次选择，强化“可体验产品”的感觉。

### 3. 交互反馈与动画

1. 切换场景时卡片、图表和链路状态进行渐变更新。
2. 当前关注子系统高亮，并同步刷新侧边说明。
3. 支持键盘可达性与 reduced motion 降级。

### 4. 离线与纯前端约束

1. 所有展示数据存放在 `src/data`。
2. 不进行 `fetch`、不调用 API、无后端依赖。
3. 依赖已有 service worker 时，新模块在离线状态下仍能正常渲染。

## 信息架构

模块采用三栏结构：

1. 左侧：场景选择器与策略摘要
2. 中部：赛道示意与趋势图
3. 右侧：遥测卡片、系统链路、关注子系统详情

移动端降级为纵向堆叠：

1. 场景选择器
2. 核心指标
3. 赛道/图表
4. 系统链路与详情

## 数据模型

新增一个静态数据文件，包含：

1. 场景元数据：`id`、`name`、`tagline`、`description`
2. 遥测数据：速度、识别率、定位置信度、控制延迟、能量回收等
3. 图表数据：若干离散点，用于渲染 SVG polyline
4. 系统阶段数据：阶段名、状态、摘要、亮点
5. 子系统详情：如感知/定位/规划/控制/车端执行器

## 技术实现

1. 使用 Astro 组件承载结构与样式。
2. 使用原生浏览器脚本完成初始化、事件绑定与 DOM 更新。
3. 将可测试的状态推导逻辑提取到 `src/utils`，避免把逻辑全写在 `.astro` 内联脚本里。
4. 图表使用原生 SVG，不引入额外图表库，控制体积与复杂度。

## 测试策略

### 单元测试

覆盖以下逻辑：

1. 场景状态推导是否正确
2. 趋势图折线点计算是否稳定
3. 本地存储读写与默认回退逻辑是否正确

### 端到端测试

覆盖以下行为：

1. 首页出现新的交互实验室模块
2. 切换场景后主标题/指标变化
3. 关注子系统切换可工作
4. 刷新后能保留用户上次场景选择

## 风险与对策

1. 风险：模块脚本过重影响首页体验。
   对策：只使用原生 DOM 和 SVG，避免引入额外运行时。

2. 风险：交互复杂导致无障碍退化。
   对策：使用按钮和 aria 属性，确保可聚焦和可读。

3. 风险：移动端布局拥挤。
   对策：首屏重点保留核心指标，详细说明折叠为纵向卡片。

## 实施步骤

1. 设计静态数据模型与状态推导工具。
2. 先写单元测试，锁定场景切换和本地持久化行为。
3. 实现首页展示组件并接入 `src/content/docs/index.mdx`。
4. 增加端到端测试覆盖交互与持久化。
5. 运行构建、单测、E2E，并根据结果修复。

## 交付标准

1. 首页新增交互实验室模块，视觉与现有风格统一。
2. 所有逻辑完全前端实现，无网络依赖。
3. 单测、构建和关键 E2E 通过。
4. 代码结构清晰，后续可继续扩展更多场景数据。
